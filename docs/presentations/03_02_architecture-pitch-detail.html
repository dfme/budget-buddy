Frage 1

Single Point of Failure?

Ja — wir haben drei. Wir haben sie bewusst akzeptiert und für jeden eine Mitigation.

🗄️ SQLite (DB-Layer) SPOF

Shared SQLite ist ein Single Writer. Bei gleichzeitigen Schreibzugriffen entsteht ein Bottleneck.

Mitigation: WAL-Modus + HikariCP max 1 Writer. Bei Wachstum: 1 Sprint für Migration zu PostgreSQL (ADR-5 dokumentiert den Pfad).

🤖 Claude API (extern) Fallback OK

Claude API down = Kategorisierung nicht möglich für ~20–30 % der Transaktionen.

Mitigation: catch AnthropicException → "Sonstiges". Import-Flow blockiert nie. Timeout gesetzt. (ADR-6)

🏗️ Monolith (ganzer JAR) Akzeptiert

Bug in report/ = gesamte App offline. Kein Fault Isolation zwischen Modulen.

Mitigation: Klare Modulgrenzen + Integration Tests mit SQLite in-memory. Für MVP: vertretbar. Upgrade-Pfad zu @Async-Workers ist dokumentiert. (ADR-4)

🔐 Auth / JWT (kein SPOF) Stateless ✓

JWT ist stateless — kein Session-Store, keine Session-Tabelle, kein Redis nötig.

Bewusste Stärke: SQLite hat keinen Schreibdruck durch Sessions. httpOnly Cookie macht Token für JavaScript unsichtbar (XSS-sicher). (ADR-7)

Frage 2

Warum diese Technologie?

Stack-Entscheide mit Begründung

🅰️
Angular 21 (TypeScript)

Component-basiert, Two-Way-Binding, Reactive Forms ideal für den Onboarding-Wizard (US-03). Signals + OnPush für performante Change Detection. Team-familiar.

☕
Java 21 + Spring Boot 3.5

Typsicher, breites Ökosystem, Industriestandard. Spring Security hat nativen JWT/Cookie-Support. Kein double für Geld — BigDecimal ist in der JVM first-class.

🗃️
SQLite + Flyway

Kein separater DB-Server nötig. Flyway-Migrationen sichern Schema-Sync im Team. Kein Concurrent-Write-Bottleneck im MVP erwartet. ADR-5 dokumentiert den PostgreSQL-Migrationsweg.

📄
PDFBox 3.x

Apache-lizenziert (kein AGPL-Problem wie iText 7). Schweizer Bank-PDFs haben einen Text-Layer — Tabula wäre falsch gewählt (Tabula ist für OCR). Apostroph-Tausender 1'234.56 wird vor dem BigDecimal-Parse normalisiert.

Frage 3

Passt das Pattern wirklich? — Modularer Monolith

📦 Warum Modularer Monolith passt

3 Entwickler · 3 Monate · MVP-Scope
Microservices verdreifachen den operativen Overhead ohne messbaren Benefit bei Nutzerzahlen im zweistelligen Bereich.

AnforderungPasst?
ACID über alle Module✅ Shared SQLite
Einfaches Deployment✅ java -jar app.jar
Integration Tests✅ SQLite in-memory
Cross-Modul-Calls✅ Direkter Methodenaufruf
Team-Onboarding✅ Eine Codebase
Horizontal Scaling⚠️ Erst bei >10k User nötig

⚠️ Risiko: "Big Ball of Mud"

Ohne konsequente Modulgrenzen kann ein Monolith zur unstrukturierten Codebasis degenerieren.

Mitigation: Kein direkter Zugriff auf Repositories anderer Module. Cross-Modul-Kommunikation nur über Interfaces (z.B. CategorizationPort). Wird im Code-Review aktiv geprüft.

🏗️ Modulstruktur — Separation of Concerns

// com.budgetbuddy
├── auth/
│   // AuthController, JWT-Config, User-Entity
├── transaction/
│   // PdfImportService, TransactionController
├── categorization/
│   // CategorizationPort (Interface!)
│   // LookupTable, ClaudeCategorizationAdapter
├── budget/
│   // SafeToSpendService, SavingsGoalService
└── report/
    // AiReportService (Sonnet 4, 1×/Monat)

🔄 Upgrade-Pfad ohne Rewrite

1
Heute: Synchroner Monolith, alles in einer JAR
2
Bei Churn: @Async + ImportJob-Entity + Status-Polling
3
Bei Wachstum: SQLite → PostgreSQL (1 Sprint)
4
Bei >10k Users: Module als Services extrahieren

⏱️ Zeitkritisch: PDF-Parsing

Apostroph-Tausender 1'234.56 + Mehrzeiligkeit im Text-Feld. Schweizer Banken: UBS, Raiffeisen, PostFinance. Fehler hier = 0 Transaktionen = kein Safe-to-Spend.

⚡ Churn-Falle: Kategorisierung

Wenn Claude API >3s braucht oder schlechte Kategorien liefert, bricht Lara den Flow ab und kommt nie wieder. Mitigation: Timeout + Fallback + manuelle Korrektur als Feature.

Frage 5

Wie testet ihr das?

🔬 Unit Tests

Was: SafeToSpendService, CategorizationService, PDF-Parser-Utils

Werkzeug: JUnit 5 + Mockito

Claude API wird über CategorizationPort gemockt — kein echter API-Call in Unit Tests.

// Beispiel
@Mock
CategorizationPort port;

@Test
void safeToSpend_returnsCorrect() {
  // BigDecimal-Arithmetik, kein double
  assertThat(result)
    .isEqualByComparingTo("342.50");
}

🔗 Integration Tests

Was: REST-Endpoints, PDF-Import-Flow, DB-Interaktionen

Werkzeug: Spring Boot Test + @SpringBootTest

Datenbank: jdbc:sqlite::memory: — kein TestContainers, kein H2 (Dialect-Mismatch!).

// application-test.properties
spring.datasource.url=
  jdbc:sqlite::memory:
// Flyway-Migrationen laufen auch
// in-memory → Schema = Prod-Schema

📋 Test-Fixtures: Echte Bank-PDFs

Raiffeisen-PDF-Fixture im Repo. Testet Apostroph-Parsing, Multiline-Text, Datum dd.MM.yyyy.

🌐 E2E / Smoke Tests

Was: Kritischer Pfad: Login → Upload → Dashboard

Werkzeug: Cypress (geplant) oder manueller Smoke Test

Wichtigster Test: PDF mit korruptem Encoding darf nie den Server zum Absturz bringen.

In CI läuft immer der Mock. Echte Claude-Calls nur in dev/staging mit echten Raiffeisen-Fixtures — manuell ausgelöst.

Frage 6

Was würdet ihr heute anders machen?

⚠️ Lokales LLM für Kategorisierung (statt Claude API)

Roher Transaktionstext verlässt heute die eigene Vertrauenszone — das ist ein nDSG Art. 16-Compliance-Gap.

Was wir ändern würden: Ollama + Llama 3.2 3B lokal für die Kategorisierung (13 Labels, kurzer Text). Claude Sonnet bleibt für den KI-Monatsbericht (komplexe Sprache).

Kosten: einmaliger Infrastruktur-Aufwand. Benefit: PII verlässt das System nie. (ADR-6 dokumentiert diesen Pfad bereits.)

⚠️ Asynchroner Import-Flow von Anfang an

Synchroner Upload-Endpoint blockiert bis zur Kategorisierung. Wenn Haiku 3s braucht bei 50 Transaktionen = Churn-Falle für Lara.

Was wir ändern würden: @Async + ImportJob-Entity von Beginn an. Frontend pollt GET /import/{jobId}/status. Lara sieht eine Progress-Bar statt einer hängenden UI.

❌ OpenBanking statt PDF-Import

Der manuelle PDF-Upload ist Churn-Falle #1. Lara bricht ab, wenn der erste Import zu umständlich ist.

Was wir anders machen würden: Früher mit Schweizer Open Banking-Anbietern (Blink, Klarna Open Banking) sprechen. CSV-Export als Zwischenlösung — einfacher als PDF-Parsing.

Realität: Schweizer Banken sind keine Early Adopter. PDF-Import bleibt MVP-Realität.

✅ Was gut war — beibehalten

✓
BigDecimal from Day 1 — kein nachträgliches Refactoring von double
✓
Flyway von Beginn an — Datenbankschema-Sync im Team ohne manuelle Skripte
✓
CategorizationPort Interface — Claude API hinter Abstraction, Austausch ohne Refactoring
✓
ADRs dokumentiert — jede Entscheidung begründet, Team spricht eine Sprache

❓ Erwartete Nachfragen

"Warum kein PostgreSQL?"

MVP-Scope, kein Concurrent-Write-Bottleneck erwartet. Migration ist 1-Sprint-Aufwand und dokumentiert (ADR-5).

"Warum kein WebFlux?"

SQLite-JDBC ist blockierend. Reactive-Wrapping würde Komplexität ohne Nutzen bringen.

"Was passiert wenn Claude API down ist?"

Fallback zu "Sonstiges" — kein API-Fehler blockiert den Import-Flow. Timeout gesetzt. (ADR-6)

"Ihr speichert Finanzdaten — wie sichert ihr das?"

JWT als httpOnly Cookie (XSS-sicher), bcrypt-Passwörter, nDSG Right-to-Deletion implementiert, Hosting Frankfurt/EU. (ADR-7, ADR-10)
