Widersprüche & Unklarheiten

Was blockiert die Implementierung?

W-1 · Kritisch

JWT Token Storage — Widerspruch im C2

C2 sagt gleichzeitig "kein Token in localStorage (HttpOnly)" und "JWT Bearer (Authorization-Header)". Beides ist inkompatibel. ADR-7 ("Logout = Client löscht Token") impliziert JS-Zugriff.

W-2 · Strukturell

SPA-Hosting — Dev vs. Prod divergieren still

ADR-0: CORS für localhost:4200. C2: SPA vom gleichen Spring-Boot-Host. Technisch konsistent, aber nirgends als zwei Phasen dokumentiert.

Kosten: Prod-Bündelung = 1 Render-Service → günstigster Pfad. Separation (CDN + API) = 2 Services, höherer Aufwand — lohnt sich erst bei eigenem CDN-Caching-Bedarf.

W-3 · Strukturell

Hosting-Plattform & nDSG-Konformität offen

Render hat keine CH-Region (nächste: Frankfurt/EU). Für Marcs Datenschutz-Skepsis fehlt ein konkretes Trust-Signal.

W-4 · Strukturell

PII-Transfer zu Anthropic API

Transaktionstext ("DIGITEC GALAXUS AG 044 913 2323") verlässt die Vertrauenszone. Kein DPA mit Anthropic. nDSG Art. 16.

W-5 · Strukturell

Fehlende Akteure im C1

Noch kein Folge-Diagramm für Admin/Operator, Backup-Ziel, Monitoring/Logging, E-Mail-Provider — alle mit Sicherheitsrelevanz.

Frage 1 für Tag 2

Wo lebt der JWT-Token? — Security-Fundament

Diese Entscheidung prägt Auth-Flow, CORS-Config, Logout-Semantik und XSS-Risiko.
Sie muss vor dem ersten Code-Commit stehen.

A) httpOnly Cookie ★ Empfehlung

Mechanismus: Browser hält Cookie; kein JS-Zugriff

✓ XSS-sicher; Anthropic Best Practice

✗ SameSite + CORS komplexer; CSRF-Schutz nötig

B) sessionStorage

Mechanismus: JS schreibt/liest; Tab-lokal

✓ Bearer-Header trivial; Tab-close = Logout

✗ XSS exponiert Token; kurzlebiger als nötig

C) localStorage

Mechanismus: JS schreibt/liest; persistent

✓ Bearer-Header trivial

✗ XSS exponiert Token dauerhaft — für Finanzdaten inakzeptabel

Empfehlung: Option A (httpOnly Cookie) ist für Finanzdaten die defensiv korrekte Wahl — auch wenn sie CSRF-Mitigation erfordert. Option B ist ein akzeptabler Kompromiss, wenn XSS durch strikte CSP mitigiert wird.
→ Output: ADR-7 vervollständigen

Frage 3 für Tag 2

Was geht an Anthropic — und unter welchem Rechtsrahmen?

Jeder Claude-Call mit Transaktionsdaten = Datenübermittlung an einen US-Dienstleister (nDSG Art. 16).

Rechtlich

DPA mit Anthropic

Anthropic bietet ein "Data Processing Agreement" für Business-Kunden an. Ohne DPA ist die Übermittlung eine Compliance-Lücke.

Technisch

Pseudonymisierung vor dem Call

Schicken wir Roh-Text ("DIGITEC GALAXUS AG …") oder reicht Betrag + normalisierter Händlertyp für Haiku 3.5?

Produktseitig

Consent in den AGB

Nutzer explizit darüber informieren, dass Transaktionsdaten zur Kategorisierung an Anthropic übermittelt werden.

Architektur

Minimalprinzip prüfen

Was braucht Haiku exakt? Möglicherweise reicht normalisierter Händlername ohne IBAN, Telefonnummer oder Betrag.

→ Output: Eintrag in ADR-6 (Kategorisierung) ergänzen
