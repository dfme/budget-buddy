Architektur-Entscheide

3 bewusste Entscheidungen

01
SQLite statt PostgreSQL
Kein separater DB-Server für MVP. Migration-Pfad in ADR-5 definiert — 1–2 Sprints, kein Rewrite.
02
Monolith statt Microservices
3 Devs, 3 Monate, unvalidiertes Produkt. Warnsignale für Extraktion in ADR-4 definiert.
03
httpOnly Cookie statt Bearer
JWT nie im JS-Zugriff → XSS-sicher. SameSite=Strict eliminiert CSRF. Kein HttpInterceptor nötig.

Kritische Fragen

Was ein CTO sofort fragt

Wie schützt ihr die Finanzdaten?
JWT als httpOnly Cookie (kein JS-Zugriff), SameSite=Strict, bcrypt-Passwörter, Server Frankfurt/EU (nDSG). SQLite speichert nur Transaktionstext + Betrag — keine Kontodaten.
Was passiert wenn Claude API down ist?
Lookup-Tabelle deckt ~70–80% ab ohne API. Bei Fehler: catch AnthropicException → Fallback "Sonstiges". Import blockiert nie.
Wie skaliert das bei 10'000 Usern?
SQLite wird Bottleneck bei concurrent writes. Upgrade-Pfad: PostgreSQL in 1–2 Sprints (ADR-5). Für MVP bewusste Entscheidung.

Risiken

Unsere 3 grössten Risiken

⏱
Churn beim ersten Upload
Wenn PDF-Import zu langsam oder fehlerhaft ist, kommen User nie wieder.
Mitigation: Sync Import + Timeout + immer Fallback
🔒
Datenleck mit Finanzdaten
Sensible Transaktionsdaten — ein Breach wäre existenzbedrohend.
Mitigation: httpOnly Cookie, bcrypt, Frankfurt/EU, nDSG
🏦
Banken bauen eigene PFM-Tools
UBS, Raiffeisen etc. könnten den Business Case über Nacht eliminieren.
Mitigation: Einfachheit des PDF-Uploads ohne Open-Banking-Onboarding
